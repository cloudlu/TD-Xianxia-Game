> 修真升维路线：**宗门 → 域 → 大陆 → 百族 → 界 → 天外**。每篇格局质变，不只是"换个地图打更硬的怪"。

### 逐篇梗概

**ch1-5 宗门篇（已实现）**：你是护阵长老，从妖狼群袭到血煞魔尊，守住宗门五次大劫。魔尊重创后透露"真正的劫难在天外"。

**ch6-10 域外篇（已实现）**：奉命调查域外魔域裂缝。苍茫域异族盘踞（沙蝎/蛮修/雾妖），最终封印裂隙主宰。域外平定，但裂缝遍布整片大陆。

**ch11-15 大陆联盟篇（已实现）**：九大宗门组建联盟。魔域远征军入侵：魔甲骑兵铁甲如山、影杀者隐遁无踪、魔蛟坚韧、魔帅万血。击败魔帅后封印大陆裂缝，但天际出现更古老的裂缝——百族大战的遗址。

**ch16-20 百族大战篇**：裂缝连通万族战场遗址，远古种族苏醒（龙族/鬼族/天妖族）。它们不是邪恶的——它们也要活下去，但代价是人类修真界的灭亡。灰色道德困境：生存本身就是错吗？最终击败远古天龙，但天魔——超越百族的存在——苏醒。

**ch21-25 界域之战篇**：天魔开始吞噬天道（世界法则本身）。灵气紊乱、修士境界跌落。所有种族被迫联合组建界域联盟。敌人是"规则扭曲"——虚空行者瞬移、混沌幼虫指数分裂、天劫化身周期全屏眩晕。最终封印天劫化身，但天道已裂——必须飞升到天外天。

**ch26-30 飞升篇**：踏入混沌原初——天魔诞生之地。最终敌人道祖魔影（50000 血/三阶段/三路分身）。天魔不是"恶"，它是天道清理过度繁衍文明的机制。结局：道祖魔影消散，天道问"你愿意成为新的天道吗？"

## 2. hpMul 曲线（章节缩放）

### 2.1 公式

```
hpMul ≈ 1 + 0.1 × (chapter - 1)    // 近似值；实际每关可微调（如 ch5 finale 手动 +0.1）
towerMul = sqrt(hpMul)              // 自动推导
bountyMul = sqrt(hpMul)             // 自动推导
```

> 实际值以代码 `levels/ch*.ts` 的 `hpMul` 字段为准，公式仅作设计参考。

### 2.2 30 章数值表

| 章 | hpMul | towerMul | 敌人基础 HP 跨度 | 玩家体感 |
|---|---|---|---|---|
| 1-5 | 1.0-1.4 | 1.0-1.18 | wolf(60) → blood_lord(8000) | 教学→有挑战 |
| 6-10 | 1.5-1.9 | 1.22-1.38 | 新梯队(200-12000) × hpMul | 需要好配装 |
| 11-20 | 2.0-2.9 | 1.41-1.70 | 更强梯队(500-20000) × hpMul | 需要满 meta |
| 21-30 | 3.0-3.9 | 1.73-1.97 | 最强梯队(1000-30000) × hpMul | 极限挑战 |

### 2.3 平衡论证

**有效 HP vs 有效 DPS**（化神境界 + 满 meta 为例）：

```
有效 DPS = 塔基础DPS × towerMul × (1 + metaMods_封顶内)
有效 HP = 敌人基础HP × hpMul

平衡条件：有效DPS / 有效HP ≈ 常数（每只击杀时间不随章节暴增）

代入：
  有效DPS = 572 × sqrt(hpMul) × 2.5 ≈ 1430 × sqrt(hpMul)
  有效HP(wolf) = 60 × hpMul

  比值 = 1430 × sqrt(hpMul) / (60 × hpMul) = 1430 / (60 × sqrt(hpMul))

  ch1: 1430/60 = 23.8（秒杀）
  ch10: 1430/(60×1.38) = 17.3（仍秒杀 wolf）
  ch30: 1430/(60×1.97) = 12.1（还是秒杀 wolf）

→ wolf 永远被秒杀。难度来自 tougher 敌人（blood_lord 8000 × 3.9 = 31200 HP）。
  31200 / (1430×1.97) = 11s/塔，5 塔 ≈ 2.2s。合理 BOSS 战时长。
```

**结论**：hpMul 的作用不是让弱怪变硬，而是让**强怪更硬**。弱怪永远是虫群（DPS 吸收器），强怪才是时间挑战。

## 5. meta 长线成长（转生系统）

### 5.1 问题

当前 meta（3 槽装备 × 5 强化 + 7 天赋 × 5 级 + 4 级 VIP + 6 皮肤）是**有限的**。约 ch10-15 全部满级后，玩家**没有更多成长空间**，30 章后半段缺乏养成驱动。

### 5.2 方案：转生（轮回再世）

通关 ch10 后解锁"转生"：重置关卡进度（回到 ch1），但保留：
- **仙魂**：永久加成货币，每转生一次获得（数量 = 通关章节数 × 10）。
- **装备/皮肤/VIP/天赋**：全部保留。
- **仙魂加成**：消耗仙魂购买永久属性（走 Modifier 管线，独立于装备），不封顶（每级递增成本）。

```
仙魂商店：
  伤害强化：每级 +5% 全体伤害，成本 = 50 × 1.2^level（不封顶）
  攻速强化：每级 +3% 全体攻速，成本 = 80 × 1.2^level
  赏金强化：每级 +5% 赏金，成本 = 60 × 1.2^level
  起始灵石：每级 +50 起始灵石，成本 = 100 × 1.2^level
```

### 5.3 转生循环

```
ch1-10 通关 → 转生（获得仙魂）→ 回 ch1（带永久加成）→ 更快通关 → ch11-20 → 再转生 → ...
```

- 每次转生后，前期关卡因永久加成变得更快（速通），玩家有"变强了"的爽感。
- 转生次数无上限，仙魂加成递减但永不封顶（成本指数增长保证不爆）。
- 仙魂加成走 Modifier 管线，与现有装备/VIP/天赋叠加，封顶仍适用（伤害 +150%）。

### 5.4 实现改动

- Progression 加 `reincarnations: number` + `soulSkills: Record<id, level>`。
- 转生纯函数 `reincarnate(p): Progression`（清空 cleared + 发仙魂）。
- 仙魂商店 = 天赋系统的复用（同为 Modifier 来源），UI 加一个"仙魂"标签。
- **引擎零改动**（仙魂走 buildMods 注入，与天赋/装备同路径）。

## 6. 无尽模式架构（已实现 v1）

### 6.1 设计目标

- 程序生成无限波次，单地图 3 路锯齿布局，**玩家最终必败**。
- 复用现有引擎（确定性、Modifier、策略），Game 加 `setHpMul()` + `addWave()` 两方法。
- 奖励回流主线：分数 → 宗门贡献 + 里程碑（首次 wave 20/40/60/80 给贡献+称号）。
- 高分存档 `endlessBest: {wave, score, date}`；累计贡献封顶 500/次。

### 6.2 奖励体系

| 奖励源 | 公式 | 说明 |
|---|---|---|
| 结算贡献 | `min(sqrt(score), 500)` | 高分多赚，封顶防刷 |
| milestone | wave 20: +150 · 40: +300 · 60: +500+称号无尽行者 · 80: +1000+永恒巡行者 | 一次性 |
| 新纪录 | +20% 结算贡献 | 鼓励突破 |
| 战斗赏金 | 走正常 bounty × bountyMul | 前 30 波经济松，后期 bountyMul 封顶 2.0 压强 |

- 程序生成无限波次，单地图，**玩家最终必败**。
- 排行榜（分数 = 总波次 × 基础分 + 击杀数）。
- 复用现有引擎（确定性、Modifier、策略），引擎改动极小（Game 加 `setHpMul` + `addWave` 两处方法）。

### 6.2 hpMul 动态曲线

```
无尽模式 hpMul(wave) = 1.05 ^ wave              // 指数增长（wave 0=1.0, wave 30=4.3, wave 70=30）
towerMul(wave) = min(sqrt(hpMul), 3.0)          // 封顶 3.0，确保后期敌人碾压
bountyMul(wave) = min(sqrt(hpMul), 2.0)         // 经济也封顶（后期经济不再增长，逼效率）
```

> ⚠️ 用**指数增长**而非线性：线性(0.05/wave)下 wave 200 仍可打，无尽变"漫长"而非"无尽"。指数让 wave 70-80 成为优秀玩家的自然极限。

| 无尽波次 | hpMul | towerMul | bountyMul | 净难度倍率 | 玩家处境 |
|---|---|---|---|---|---|
| 0-10 | 1.0-1.6 | 1.0-1.27 | 1.0-1.27 | 平衡 | 轻松 |
| 20 | 2.7 | 1.64 | 1.64 | 1.65× | 有挑战 |
| 30 | 4.3 | 2.08 | 2.0(封顶) | 2.07× | 困难 |
| 50 | 11.5 | 3.0(封顶) | 2.0(封顶) | 3.83× | 极限 |
| 70 | 30.1 | 3.0(封顶) | 2.0(封顶) | 10× | **几乎不可能** ✓ |
| 100 | 131 | 3.0(封顶) | 2.0(封顶) | 43.7× | **必败** ✓ |

**双重封顶**（towerMul 3.0 + bountyMul 2.0）是关键设计：
- towerMul 封顶 → 塔伤害增速远不如敌人血量 → 后期每只怪打不死。
- bountyMul 封顶 → 经济不再增长 → 无法靠堆塔数量弥补 → 逼策略极限。
- 自然终点出现在 wave 70-80 左右，是合理的无尽分数目标（优秀玩家极限）。

### 6.3 波次生成规则

```
每 5 波为一轮：
  wave % 5 == 0: BOSS 波（召唤 1 只精英 + 杂兵）
  wave % 5 == 1-4: 普通波（从已解锁敌人池随机选 2-3 种混搭）
  每 10 波：敌人池扩充（解锁下一梯队的敌人类型）
```

- 敌人池随波次解锁新类型（ch1-5 敌人 → ch6-10 → ...），模拟"章节推进"。
- 路径：固定使用一张多路径地图（3 路），波次随机分配敌人到不同路径。
- 波次间隔：固定 10s prep（不随波次缩短，给玩家喘息）。

```ts
  };
}

// Game 运行时支持动态 hpMul + 动态波次
// 方案 A（推荐）：Game 暴露 setHpMul(n) + addWave(wave) 方法
// 方案 B：每次波次清空后重新生成 LevelConfig（更干净但重建 Game）
```

**推荐方案 A**：Game 加两个方法：
- `setHpMul(n)`：更新 this.hpMul + this.towerMul（影响后续 spawn 的敌人 + effectiveStats）。
- `addWave(wave)`：向 this.level.waves 追加一条波次（影响后续 waveIndex）。

引擎其余部分（spawnEnemyAt、effectiveStats、updateWave）**完全不改**。

### 6.5 分数与排行

```
分数 = Σ(波次号 × 100) + Σ(击杀数 × 10) + 剩余血量 × 50
```

分数存入玩家档案（profile.endlessBest），选关界面显示历史最高分。

## 8. 重玩与活动系统

### 8.1 难度模式（三档，独立命名）

现有所有关卡可套用三种难度倍率，**零额外关卡设计**——纯运行时乘数。不使用境界名（避免与塔境界系统混淆）：

| 难度 | 敌人血量倍率 | 赏金倍率 | 定位 |
|---|---|---|---|
| **简单** | ×0.8 | ×1.1 | 休闲/剧情玩家通关用 |
| **普通** | ×1.0 | ×1.0 | 默认平衡 |
| **困难** | ×1.5 | ×0.85 | 老手挑战 |

> 为什么只有三档：90 关 × 3 星已是巨量内容，不需要难度模式凑重玩性。硬核玩家有**无尽模式**（wave 70+ 比任何固定难度都难）。三种难度覆盖"能通关 → 有挑战 → 极限"三段需求即可。

**解锁规则**（难度顺序递进）：
```
简单：关卡链式解锁（和当前一样：通关 ch1-l1 → 解锁 ch1-l2）
普通：某关在简单通关后 → 该关解锁普通（仍需链式推进）
困难：某关在普通通关后 → 该关解锁困难（仍需链式推进）
```

**星级隔离**：`cleared` key 格式为 `${levelId}:${difficulty}`，每难度独立记录星级。

实现：`isUnlocked(manifest, index, p, difficulty)` 按难度独立判断；`recordResult(p, levelId, difficulty, stars)` 按难度写星级。`withDefaults` 将旧单 key 存档迁移为 `key:normal`。

### 8.2 活动关卡（限时/节日）

利用 manifest 的 `availability` 字段控制可见性：

```jsonc
{ levelId: 'event-spring-1', chapterId: 'event', chapterTitle: '春节活动 · 年兽来袭',
  unlock: { clear: 'ch5-l3' },
  availability: { type: 'event', from: '2026-02-01', until: '2026-02-15' } }
```

- **节日活动**：春节·年兽、中秋·月华、周年庆·限定 BOSS，配套限定皮肤/法宝奖励。
- **限时挑战**：特殊限制条件关卡（如"只能用剑修塔""30 秒内通关""零漏怪"），给挑战型玩家。
- **活动商店**：活动代币兑换限定奖励（独立于常驻 meta）。
- 活动关卡不占用主线解锁链，独立 manifest 分组。

### 8.3 挑战模式（特殊限制）

每关可附加可选挑战条件，通关给额外星/成就：

| 挑战 | 条件 | 奖励 |
|---|---|---|
| 速通 | X 秒内通关 | 额外贡献 |
| 零漏 | 不漏任何敌人 | 额外星 |
| 单流派 | 只用一种塔 | 限定称号 |
| 禁升级 | 不升级任何塔 | 限定皮肤碎片 |

实现：LevelConfig 加可选 `challenges?: Challenge[]`，Game 追踪挑战状态，结算时检查。

## 9. 架构兼容性检查

| 系统 | 当前状态 | 30 章/无尽需要改吗？ |
|---|---|---|
| TowerBehavior 策略注册表 | projectile/pierce/aoe/chain/aura | 加新 behavior = 加 strategy 类，不改 Game ✓ |
| EnemyConfig ability 字段 | fly/shield/lifesteal/knockback/stealth/split/bossAbility | 加新 ability = 加 ability 处理器 ✓ |
| Modifier 管线 | 多源合并 + 按 stat 封顶 | 新境界/新塔自动走管线 ✓ |
| hpMul 缩放 | 已实现（spawnEnemyAt + effectiveStats） | 无尽模式只需 setHpMul() ✓ |
| WaveDirector | 固定波次 | 无尽模式只需 addWave() 或动态 level.waves ✓ |
| SaveRepo | per-profile localStorage | 无尽分数存 profile.endlessBest ✓ |
| maxTowerLevel | per-level 配置 | 新境界只需在 towers.ts levels 追加 + maxTowerLevel 调高 ✓ |

**结论**：当前架构兼容 30 章 + 无尽模式的**配置/策略层面**（OCP 保证加塔/敌人/境界不改 Game）。无尽模式需 Game 加两处小方法（`setHpMul` + `addWave`）。部分 🔴 子系统级新机制（物理免疫/反弹/毒雾/混乱）需要引擎新增子系统，应分阶段实现、先做 🟢 处理器级机制。

## 10. Review 备忘（供 solid skill review 用）

本文档的平衡设计基于以下数学约束，review 时应验证：

1. **towerMul = sqrt(hpMul)**：保证经济-战力平衡（总 DPS 增长匹配 HP 增长）。
2. **bountyMul = sqrt(hpMul)**：赏金适度跟上（比 HP 涨得慢，经济逐渐变紧）。
3. **无尽 towerMul 封顶 3.0**：保证有自然终点（玩家必败）。
4. **不缩放 armor/speed**：避免多轴复合爆炸（设计文档 §5.1 的教训）。
5. **Modifier 封顶绝对值**：meta 加成封顶不随章节/hpMul 变（防 P2W 爆炸）。
6. **弱怪永远被秒杀**：hpMul 对弱怪(wolf 60hp)几乎无影响；难度来自强怪(BOSS 8000+hp)的选型。
7. **新境界 ×2 跃升**：每个新境界 DPS 翻倍，与 hpMul 的温和增长(×1.1/章)形成"境界突破"的爽感节点。
