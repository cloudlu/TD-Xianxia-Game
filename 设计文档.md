## 0. 设计目标（五大可扩展性约束）

本文档所有架构决策都服务于以下五个目标，任何新功能落地前都应回头校验：

1. **剧情可扩展**：新增章节/关卡/剧情 = 加 JSON 文件，零代码。
2. **塔和敌人可扩展**：数值型新增 = 加 JSON 条目；**新行为机制**需加一个 behavior/ability 处理器（见 §4.4、§5.4）。
3. **平衡易调整**：所有数值在配置文件中，提供"平衡旋钮"，策划可独立调节。
4. **前后端可分离**：核心逻辑与数据访问通过"仓储接口（Repository）"隔离。**注**：Repo 实现可无缝替换，但**真正的服务端权威防作弊依赖确定性模拟**（见 §1.4、§12.2），需在设计期就奠定，否则联网阶段需返工。
5. **通用功能易扩展**：充值、VIP、装备、皮肤、加成等通过统一的 **Modifier（修饰器管线）** 接入，战斗结算逻辑不因新功能而改动（运算规范见 §9.5）。

## 1. 技术架构

### 1.1 分层总览

```
┌─────────────────────────────────────────────┐
│  表现层 UI（DOM/Canvas）                      │  ← 可替换：Web版/未来终端版
├─────────────────────────────────────────────┤
│  游戏引擎 Engine（纯逻辑，无 UI 依赖）         │  ← 核心，确定性 + 固定步长
│  · 主循环 · 战斗 · 寻路 · 波次 · 数值结算      │
├─────────────────────────────────────────────┤
│  系统层 Systems（玩家成长功能）               │  ← 装备/VIP/皮肤/存档
├─────────────────────────────────────────────┤
│  数据层 Data                                  │  ← 全部 JSON 配置
│  · Registry（按id查询） · Modifier管线        │
├─────────────────────────────────────────────┤
│  仓储层 Repository（接口）                    │  ← 前后端分离的关键
│  · SaveRepo · ProfileRepo · IAPRepo · ...    │
├──────────────────────┬──────────────────────┤
│  本地实现（localStorage） │ 远程实现（HTTP/SDK，未来）│
└──────────────────────┴──────────────────────┘
```

### 1.2 前后端分离的关键：仓储接口（Repository Pattern）

所有"数据从哪来"的问题都收敛到接口。引擎只依赖接口，不依赖具体实现。

```ts
interface SaveRepo     { load(): SaveData; save(d: SaveData): void; }       // 存档
interface ProfileRepo  { getProfile(): PlayerProfile; update(p): void; }    // 玩家档案
interface IAPRepo      { getProducts(): Product[]; purchase(id): Promise<Receipt>; } // 充值
interface LeaderboardRepo { submit(score): void; top(n): Score[]; }         // 排行榜
interface AuthRepo     { login(): Account; }                                // 账号
interface TelemetryRepo { log(event): void; }                               // 遥测（见 §14）
```

- **单机模式**：全部用 `Local*Repo`（localStorage 实现）。
- **联网模式**：换成 `Remote*Repo`（HTTP/WebSocket 实现），引擎与系统层接口调用不变。
- 切换只需在启动时注入不同实现（依赖注入）。
- ⚠️ **注意**："换实现引擎不动"指的是**数据访问层**；真正的服务端权威防作弊是另一回事，见 §12.2。

### 1.4 确定性引擎与固定步长（S2 / N5）

**为什么必须现在定**：可变步长 `update(dt)` 有两个问题——① 卡顿后大步长会让敌人"瞬移"穿过塔射程而不被打到（隧穿）；② 联网防作弊需要服务端能复算战斗结果，非确定性模拟无法复算。这两个问题在 P1 写代码时就要规避，否则 P6 联网要推翻重写。

**设计约定**：
- **固定步长模拟**：逻辑步长 `fixedDt = 1/30s`，主循环用累加器（accumulator）消化实际帧时间，每攒满一个 fixedDt 推进一次逻辑；渲染层对逻辑状态做插值，保持画面流畅。
- **种子化随机**：所有随机（暴击、闪避、分裂、掉落）走 `PRNG`（如 mulberry32），种子来源于关卡种子 + 玩家操作序号。同种子 + 同操作 → 同结果。
- **关键数值整数化**：灵石、赏金、血量等经济/血量用整数运算；护甲减伤等中间量不可避免浮点，但**结算点四舍五入到整数**，保证跨平台一致。
- **禁止**在引擎层使用 `Math.random()`、`Date.now()`、浮点依赖的相等比较。

> 这样设计后，未来联网可走"客户端跑战斗 + 上报操作流水 + 服务端按种子重放校验"的真防作弊路线（§12.2），无需重写引擎。

## 3. 数据驱动的配置系统

**核心原则**：用 ID 引用，不用继承。引擎通过 `Registry` 按 id 查询配置对象，新增内容只加 JSON。

```ts
class Registry {
  tower(id): TowerConfig;       // 查塔
  enemy(id): EnemyConfig;       // 查敌人
  chapter(id): ChapterConfig;   // 查章节
  equipment(id), skin(id), vip(level) ...
}
```

### 3.1 配置校验

`ConfigLoader` 加载时做 schema 校验（必填字段、数值范围、id 唯一性、引用完整性、behavior/ability 名是否已注册），配置错误在启动时报错而非运行时崩溃。

### 3.2 扩展性的两类区分（重要）

| 扩展类型 | 是否零代码 | 例子 |
|---|---|---|
| **数值/组合扩展** | ✅ 零代码（加 JSON 条目） | 新增一个"火鸦符"塔，复用 `projectile` behavior |
| **新行为机制扩展** | ❌ 需加处理器 | 新增"召唤类塔"需注册新 `summon` behavior 处理器 |

文档中所有"零代码"承诺均指第一类。第二类已在引擎预留 behavior/ability 注册表（§4.4、§5.4），新增处理器是局部改动，不影响其它内容。

## 4. 塔系统（含完整数值）

### 4.1 设计基准（S3 修订）

- **基准效率**：`1 灵石 ≈ 5 单体 DPS`（单体均衡塔的标准）。
- **功能性塔**（减速/经济/AOE）效率略低，价值在机制。
- ⚠️ **宽度 vs 深度张力**（修订）：升级效率**不应严格单调递增**。升级一座塔提升 DPS，但**射程覆盖有限**，面对多路线/分散刷怪时单塔守不住；铺多座低级塔覆盖广但单点 DPS 不足、难破厚血怪。两者必须各有适用场景，避免"养一座化神塔就赢"的单解。
- 所有数值在 `config/towers.json`，可调。

**已注册 behavior**（引擎实现这些处理器，配置引用即可）：

| behavior | 说明 |
|---|---|
| `projectile` | 飞行物命中单体 |
| `hitscan` | 即时命中单体 |

新增 behavior = 在 CombatSystem 注册一个处理器 + 配置引用，其它塔不受影响。

### 4.5 五行 / 功法克制（可选深度，可后置）

- 火法克"木灵虫群"、冰符控"蛮兽冲锋"、重剑打"高护甲魔甲"、雷法克"水属性妖修"。
- 实现：敌人 `element/tags`，塔 `bonusVs`，结算乘克制系数（默认 1.5×）。
- Phase 1 可先关闭，用 `enabled: false`。

### 4.6 战斗规则（M2 新增）

塔防标配机制，引擎与配置均已预留：

**① 目标选择 AI（Targeting）**
每座塔有 `targetPolicy`，玩家可在战斗中切换：
- `first`（最前/最接近出口）、`last`（最后/刚出生）
- `strongest`（最高血量）、`weakest`（最低血量）、`nearest`（最近）

**② 加速与暂停**
1× / 2× / 3× / 暂停。实现上 = 调整固定步长推进频率（§1.4），暂停冻结逻辑帧。加速不破坏确定性（只是单位时间内推进更多步）。

**③ 提前召唤下一波**
玩家可在当前波未清完时手动召唤下一波，获得**提前召唤奖励**（如 +20% 该波赏金）。既是节奏控制，也是经济杠杆（高手用它滚雪球）。

**④ 出售规则**
- 波次间隙：按 `sellRatio`（默认 0.6）返还。
- 波次进行中：允许出售但**返还折半**（0.3），避免频繁拆建刷经济。

**⑤ 三星评定标准**（§8.6 详述，此处先给定规则）
基于通关时**剩余血量**评定：满血 3 星、剩余 ≥60% 两星、通关即一星。星级解锁 meta 奖励（§10）。

## 5. 敌人系统（含完整数值）

### 5.1 单位约定（S4 修订）

- **速度**：格/秒。典型路径长 16 格，1.0 速度 = 16 秒走完全程。
- **护甲**：`实际伤害 = 攻击力 × (100/(100+护甲))`。护甲 40 → 只受 71% 伤害。
- ⚠️ **赏金独立配置**（修订）：赏金**不再强制 = 血量/10**，改为 `bounty` 字段独立配置，遵循四条规则：
  1. **普通怪** bounty ≈ 血量/10（保持基准）；
  2. **高护甲/护盾/回血怪** bounty 按"有效血量"上浮，补偿玩家更高投入：
     `bounty = round( 有效血量/10 × (1 + armor/200) )`，**有效血量 = hp + shield + 回血折算**（傀儡 400hp/40甲 → 48≈50；魔修喽啰 250hp+50盾 → 30；血修按回血折算上调）。公式仅作参考，策划可直接手填覆盖；
  3. **BOSS** bounty = 0（击杀不给关卡内灵石），改为通关结算时发放 **`bossClearContribution`（宗门贡献，归 meta，见 §10）**，彻底不进关卡内即时经济循环，防止中段 BOSS 打爆后续平衡；
  4. **分裂子体** bounty = 0 或大幅降低（防分身妖刷钱）。

**小头目 vs BOSS 的区分（T11）**：
- **小头目**（山猪王、魔修统领等）：`isBoss: false`，是"血量偏高的普通敌人"，有 bounty、走正常波次经济、**不给** `bossClearContribution`；
- **真 BOSS**（九尾天狐、烛龙、魔尊血煞）：`isBoss: true`，bounty=0、通关发 `bossClearContribution`、带 BOSS 技能（受 §5.4 护栏约束）。

### 5.4 特殊机制实现预留

每种机制怪的行为通过 `abilities` 字段声明，引擎按能力名查找处理器（策略模式）。新增机制 = 加一个 ability 处理器 + 配置引用（第二类扩展，见 §3.2）。

| ability | 行为 |
|---|---|
| `shield:N` | 护盾层，先破盾 |
| `lifesteal:ratio` | 受击回血 |
| `split:N` | 死亡分裂（子体赏金单独配置） |
| `stealth` | 隐身，需破隐塔 |
| `fly` | 飞行，免疫近战 |
| `explode` | 死亡爆炸 |
| `reflect` | 移动反伤 |
| `knockback_tower` | 经过塔时**击退**塔（塔短暂瘫痪 2s，不销毁，可恢复） |
| `destroy_tower` | 撞到塔时**摧毁**塔（玩家损失该塔，按 `sellRatio` 返还灵石，可在原位或别处重建） |
| `charm` | BOSS 魅惑（见下方护栏） |
| `eclipse` | BOSS 日蚀致盲（见下方护栏） |

**控场技能平衡护栏（R4，防软锁）**：
所有"使塔失去作用"的技能必须满足以下约束，避免玩家被无解恶心：
- **范围受限**：只影响技能作用圈内的塔（如祭坛范围/全屏需配置明示），不存在"凭空影响全场且无对策"；
- **可视预警**：触发前 2-3s 给明显视觉预警（地面红圈/天色变暗），玩家可提前应变（提前召唤、调整目标策略）；
- **持续时间上限**：单次 ≤ 3s；
- **冷却免疫**：同一塔被作用后有 6-8s 免疫，防连续软锁；
- **对策存在**：致盲（日蚀）/魅惑只削弱**需要命中判定**的攻击（`projectile`/`hitscan`/`pierce`/`chain`）；**AOE 区域伤害、`aura` 光环、`dot` 持续伤害**不依赖命中判定，**不受日蚀/魅惑影响**，给玩家留明确反制手段（T6）。

**撞塔类敌人规则（R3）**：
- 蛮牛默认用 `knockback_tower`（击退，温和）；高难模式或精英变种可配 `destroy_tower`（摧毁，严厉）；
- 被摧毁的塔按 `sellRatio` 返还灵石，玩家可重建；
- 玩家对策：用减速/冰冻塔先控住蛮牛，或把核心塔放在路径拐角后方（蛮牛直线冲撞时不易触及）；
- **连环击退护栏（T9）**：同一塔被击退后有 **6s 免疫**，防止一群蛮牛连续通过导致某塔被永久瘫痪。

## 7. 平衡性系统

### 7.1 难度由三层叠加（每层独立可调）

**① 关卡内波次递增**
```
第N波敌人血量 = 基础血量 × (1 + globalHpGrowth × (N-1))   // globalHpGrowth 默认 0.15
第N波数量     = 基础数量 + ceil(N/2)
```

**② 章节间基础值递增**

| 章 | 妖狼基础血量 | 引入新怪 | 建议塔均境 |
|---|---|---|---|
| 1 | 60 | 妖狼/山猪 | 炼气~筑基 |
| 2 | 110 | 蛮牛/毒蟾/蝙蝠 | 筑基~金丹 |
| 3 | 180 | 魔修/血修/傀儡 | 金丹~元婴 |
| 4 | 280 | 隐身狐/分身/BOSS | 元婴 |
| 5 | 400 | 终极BOSS综合 | 元婴~化神 |

**③ 全局难度倍率**（玩家自选）

| 难度 | 血量倍率 | 赏金倍率 |
|---|---|---|
| 炼气(简单) | ×0.8 | ×1.1 |
| 筑基(普通) | ×1.0 | ×1.0 |
| 金丹(困难) | ×1.4 | ×0.9 |
| 元婴(噩梦) | ×1.9 | ×0.8 |

### 7.2 平衡演算

**场景**：第1章第1关第1波，8只妖狼，开局300灵石。
- 建 3 飞剑(240) + 1 符箓(60) = 4 塔，总 DPS = 78。
- 妖狼血60，速度1.2，射程2.5覆盖约4格路径 → 在射程内 `4/1.2 ≈ 3.3秒`。
- 单飞剑 3.3 秒输出 = 66 > 60 ✓ 路径内可杀一只。
- 总 DPS 78 × 3.3s = 257，足够杀 4 只；射程重叠后首波轻松守住 ✓。
- 进账：8×6 赏金 + 50 清波 = 98 灵石。

**反例（难度梯度）**：第3关魔修喽啰 250血+50盾。炼气飞剑 DPS20 需 15 秒，但路径覆盖仅 3.3 秒 → **必须升级到金丹或换雷法/重剑**，逼玩家成长 ✓。

**宽度 vs 深度张力验证**（S3）：化神飞剑 DPS 260、射程 3.6 格，单塔只能覆盖一段路径。面对第4章**双路线 + 分散刷怪**，一座化神塔守不住两路，玩家必须**分兵铺塔**或用聚灵阵增益多塔——证明铺塔流有存在价值 ✓。

### 7.3 平衡"旋钮"一览

| 旋钮 | 作用 | 位置 |
|---|---|---|
| `startStones` | 开局灵石松紧 | 关卡JSON |
| `globalHpGrowth` | 波次血量增长 | 章节JSON |
| 难度倍率 | 整体难度 | manifest |
| 塔 DPS/费/射程/behavior | 塔强弱 | towers.json |
| 敌人血/速/甲/bounty | 怪强弱 | enemies.json |
| `clearBonus`/赏金系数 | 经济松紧 | 波次/敌人 |
| 升级边际效率曲线 | 铺塔 vs 升级平衡 | towers.json levels |

> 平衡核心公式：**塔 DPS × 路径覆盖时间 ≥ 敌人血量** 才能守住；**升级 DPS 收益 vs 多塔覆盖收益**决定战术形态。

## 8. 章节 / 关卡 / 剧情系统（可扩展核心）

### 8.1 数据驱动的章节

```
config/chapters/
├── _manifest.json        ← 章节顺序、解锁条件、难度倍率
```

## 9. 玩家成长与加成系统（Modifier 管线）

### 9.1 统一加成管线（可扩展性的核心）

所有"加成"——装备、皮肤、VIP、科技树、活动 buff——都通过同一条 Modifier 管线注入战斗结算，**结算逻辑不因新功能而改动**。运算规范见 §9.5。

```ts
// 任意一次伤害/金币结算 = 基础值 经过 Modifier 管线（按 §9.5 公式结算）
function computeDamage(base, ctx) {
  return Modifier.resolve(base, ctx, [
    equipmentMod(player.equipment),
    skinMod(player.skin),
    vipMod(player.vip),
    techMod(player.tech),
    eventBuffMod(player.buffs),
  ]);
}
```

### 9.2 装备 / 本命法宝（equipment.json）

```jsonc
{
  "green_lotus_sword": { "name": "青莲剑", "slot": "weapon", "rarity": "epic",
    "mods": [{ "stat": "swordDmg", "op": "mul_pct", "value": 0.15 }] },
  "lifire_fan": { "name": "离火扇", "slot": "weapon", "rarity": "legend",
    "mods": [{ "stat": "fireRange", "op": "add", "value": 0.5 }] }
}
```

### 9.5 Modifier 运算规范（S1 新增——必读）

为避免 `add`/`mul` 混用导致顺序敏感、数值爆炸，统一定义三类操作与**固定结算公式**（结算顺序由公式决定，与 mod 在数组中的位置无关）：

**三类操作**：

| op | 含义 | 叠加方式 | 用途 |
|---|---|---|---|
| `add` | 扁平加成 | 所有 `add` 求和 | 离火扇 +0.5 射程、VIP +50 开局灵石 |
| `mul_pct` | 百分比加成 | **加法叠加**（Σ value） | 绝大多数加成（装备/皮肤/VIP 伤害/赏金%） |
| `mul` | 真实乘法 | **乘法叠加**（Π value） | 仅特殊场景，慎用（默认不开放给配置） |

**统一结算公式**：

```
final = (base + Σadd) × (1 + Σmul_pct) × Πmul
```

- 同一 `stat` 的所有 `add` 先求和，所有 `mul_pct` 加法求和，所有 `mul` 乘法求积；
- 顺序固定为 `(base+add) → mul_pct → mul`，**与 mod 注册顺序无关**。

**举例**：base=100，装备 swordDmg `mul_pct 0.15`，VIP swordDmg `mul_pct 0.10`，科技 swordDmg `mul_pct 0.20`：
```
final = 100 × (1 + 0.15+0.10+0.20) = 100 × 1.45 = 145   // 加法叠加，可控
```
（若误用乘法叠加：1.15×1.10×1.20 = 1.518，难以预测且易膨胀。）

**跨 stat 组合规则（T1，关键）**：
§9.5 只定义了**同一 stat 内**加法叠加。但一座塔常同时受多个同性质 stat 影响（如飞剑修士同时吃 `swordDmg` 流派乘子和 `dmg` 通用乘子）。为避免跨 stat 乘法突破封顶、保证数学可控，规定：
- 结算时按目标**适用的 stat 集合**，把所有"伤害类"乘子（`dmg`/`swordDmg`/`bowDmg`/`spearDmg`/`magicDmg`/`critDmg` 折算等）**先合并为一个有效伤害乘子**：`effectiveDmgMul = Σ(所有适用伤害类 stat 的 Σmul_pct)`；
- 即流派乘子与通用乘子**加法合并**，而非相乘；
- 合并后的 `effectiveDmgMul` 统一受"伤害类 +150%"封顶约束；
- 非伤害类（攻速/射程/经济/减速）各自独立按 stat 封顶。

举例（飞剑修士）：`swordDmg Σ=0.35`、`dmg Σ=0.27` → 不是 `1.35×1.27`，而是 `effectiveDmgMul = 0.62`，`final = base × 1.62`，受 +150% 封顶保护。

**上限（防膨胀，按 stat 分级，R5）**：
为避免后期 meta 天赋+装备+VIP 叠加把难度打穿，封顶**按 stat 分级**（而非统一 +200%）：

| stat 类别 | `Σmul_pct` 封顶 | 说明 |
|---|---|---|
| 伤害类（dmg/swordDmg/...，合并后） | +150% | 战斗力天花板（T1 合并后封顶） |
| 经济类（bountyMul/...） | +80% | 防经济爆裂 |
| 攻速/射程类 | +60% | 防机制失衡 |
| `add` 类 | ≤ 基础值的 1 倍 | 防极端溢出 |
| 暴击率（crit） | ≤ 0.6（绝对值） | 防必暴（T7） |
| 冷却缩减（cooldown） | ≥ -40% | 防技能无CD（T7） |

- 超过上限的部分截断，并写遥测告警（§14）便于发现平衡异常；
- **难度联动校准（T13）**：高难度（金丹/元婴）的敌人倍率**由策划参照"满 meta 玩家强度"手动配置**（非系统自动），确保付费/长线玩家在噩梦难度仍有挑战。

**实现**：`Modifier.resolve(base, ctx, sources)` 内部按 stat 分组、分类求和、套公式、套上限，返回最终值。新增加成来源只需产出 mod 数组，无需改 resolve。

## 12. 联网 / 账号系统

### 12.1 渐进式联网策略

| 阶段 | 形态 | 实现 | 防作弊强度 |
|---|---|---|---|
| A | 纯单机 | 全部 Local*Repo | 无需 |
| B | 云存档 | SaveRepo 换 Remote + 本地缓存 | 弱（存档同步） |
| C | 账号 + 充值 | AuthRepo + IAPRepo 换 Remote | 中（敏感数据服务端权威） |
| D | 排行榜/社交 | LeaderboardRepo + 好友/公会 | 中 |
| E（可选） | 服务端权威战斗 | 操作流水重放校验 | 强 |

切换 = 启动时注入不同 Repo 实现，**数据访问层零改动**。

### 12.2 防作弊边界（S2 修订——诚实界定）

**关键认知**：客户端永远不可信。不同阶段能做到的防作弊强度不同，文档不再承诺"配置校验即可防刷"：

- **C 阶段（账号+充值）**：仙玉/道具/VIP 等**敏感数据服务端为唯一真源**，客户端只读。这一层是扎实的，能挡住直接改存档刷仙玉。
- **战斗结果**：客户端跑战斗、上报"通关/星级"。服务端能做的**轻度校验**：通关时长下限、星级与血量一致性、关卡解锁前置。**这挡不住"假装正常通关骗奖励"**——因为战斗逻辑在客户端，服务端无法仅凭结果判断真伪。
- **E 阶段（可选，强防作弊）**：服务端用**确定性模拟**（§1.4）重放客户端上报的操作流水，校验最终状态是否一致，不一致即判作弊。**正因为引擎从 P1 就是确定性的**（固定步长 + 种子 PRNG + 整数结算），这条路才可行，无需重写引擎。

> 结论：P1~P5 单机/mock 阶段不操心；P6 上线先做到 C 阶段（敏感数据服务端权威 + 战斗轻度校验）；若日后作弊严重，再上 E 阶段重放校验。**确定性引擎是这条演进路线的入场券，现在就要埋好。**

## 13. 存档系统

- **关卡内进度**：可中断续战（保存当前波次/塔布局/灵石 + 关卡种子，便于确定性续算）。
- **玩家档案**：跨关卡的养成（VIP/装备/皮肤/通关进度/Meta）。
- 通过 `SaveRepo` 抽象，本地用 localStorage，联网后换云端。
- 存档做版本号（`schemaVersion`），便于未来迁移。
- **云存档冲突策略**（联网后）：以"最后修改时间"为准 + 保留两端历史副本供玩家选择；关键经济字段（仙玉/贡献）以服务端为准，不本地合并。
- localStorage 是同步阻塞且有 ~5-10MB 上限：存档写入做 **debounce + 异步封装**，避免卡帧。
