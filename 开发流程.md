# 仙侠塔防 · 开发流程规范

> 定义本项目所有功能的开发流程。每步必须遵守，保证架构可扩展、代码可维护、文档与实现一致。

## 1. 总流程（五步法）

```
需求 → 设计 Review → 文档更新 → 实现 → 验证
  ↑                                            │
  └────────────────────────────────────────────┘
```

**不可跳步**。设计未 review 不许写代码；文档未更新不许合并。

---

## 2. 设计阶段（按架构 skill）

### 2.1 触发条件
任何涉及以下之一的功能：
- 新系统（新机制/新敌人能力/新塔行为/新 UI 模块）
- 数据结构变更（Progression/LevelConfig/EnemyConfig/TowerConfig schema）
- 引擎改动（Game.ts / CombatSystem / WaveDirector）
- 平衡调整（塔数值/敌人数值/经济参数/章节缩放）

### 2.2 设计输出（写在文档里，不是口述）

| 内容 | 位置 |
|---|---|
| 系统架构描述 | 涉及哪些模块、新增哪些文件 |
| 配置 schema | 字段名/类型/默认值/示例 |
| 引擎改动点 | 具体到函数/方法/文件 |
| 数值公式 | 带演算验证 |
| 平衡论证 | 与现有系统的兼容性推演 |
| 实现复杂度 | 🟢配置/🟡引擎小改/🔴新子系统 |

### 2.3 Review 检查清单（game-design-architecture skill 的五大可扩展性）

```
[ ] 剧情可扩展：加章节是否只加数据？（不加代码）
[ ] 塔和敌人可扩展：数值类是否纯配置？新机制是否走 registry/strategy？
[ ] 平衡易调整：数值是否在配置文件中？是否提供了平衡旋钮？
[ ] 前后端可分离：数据访问是否走 Repository 接口？
[ ] 通用功能可扩展：加成是否走 Modifier 管线？
```

### 2.4 发现的问题分级

- 🔴 **严重**：会返工/会崩平衡/承诺做不到。**不修不许进入实现**。
- 🟡 **重要**：影响完整性/体验。应修。
- 🟢 **次要**：建议。可后置。

## 3. 实现阶段（按 SOLID skill）

### 3.1 五条原则逐项过

| 原则 | 检查问题 |
|---|---|
| **S**RP 单一职责 | 这个类/模块只有一处变的理由吗？ |
| **O**CP 开闭原则 | 加新内容需要改已有代码吗？策略/注册表/配置就够了吗？ |
| **L**SP 里氏替换 | （本游戏以函数式为主，此条较少触发，但不制造隐藏契约） |
| **I**SP 接口隔离 | 接口是否只暴露调用方需要的方法？ |
| **D**IP 依赖倒置 | 引擎依赖抽象（ConfigLookup/CombatContext/SaveRepo）还是具体实现？ |

### 3.2 代码规范

```typescript
// ✅ 正确：配置驱动，纯函数
export function computeStars(lives: number, startLives: number): number { ... }

// ✅ 正确：策略注册表，不改 Game
class ProjectileStrategy implements AttackStrategy { ... }

// ✅ 正确：Repository 抽象
export interface SaveRepo { load(): Progression; save(p: Progression): void; }

// ❌ 错误：硬编码数值
if (towerId === 'flying_sword') { dmg = 20; }

// ❌ 错误：向引擎散落新逻辑
// Game.fire() 里加 if (behavior === 'new_type') { ... }
```

### 3.3 测试要求

- **纯函数必须有测试**（combat、economy、progress、titles、Modifier管线）
- 配置变动必须同步更新测试预期值
- 新机制加新测试（参考 `progress.test.ts` 的模式）
- 运行 `npm test` 全绿后方可进入验证阶段

---

## 4. 文档更新（不可跳步）

### 4.0 ⚠️ 必须——实现开始前立即写下需更新的文档清单

**在写第一行代码之前**，根据改动范围列出"需要更新哪些文档"并记在对话中。实现完成后对照清单逐条划掉。不允许 typecheck 通过就直接认为完成。

示例：
```
改动：talents.ts 新增 fire_mastery + thunder_mastery；equipment.ts spirit_flame 改 stat
→ 需更新文档：[ ] 装备与皮肤设计.md（装备表/stat 名变更）
               [ ] 设计文档.md（如有提到天赋数量）
```

### 4.1 更新规则

| 文档 | 什么情况下必须更新 |
|---|---|
| **设计文档.md** | 新增/变更系统架构、机制、平衡公式；实现现状小节 |
| **内容设计.md** | 新增/变更章节、关卡、敌人、塔、叙事 |
| **装备与皮肤设计.md** | 新增/变更装备、皮肤词条 |
| **README.md** | 影响运行方式、部署方式、核心玩法 |

### 4.2 不得出现的情况

- ❌ 代码实现了但文档还写"未实现"
- ❌ 数值改了但文档公式是旧值
- ❌ 删了敌人/塔但文档表格还列着
- ❌ 文档里引用了已删除的文件名

---

## 5. 验证阶段

### 5.1 必须通过

```
npm run typecheck    # TypeScript 零错误
npm test             # 全部单元测试通过
npm run build        # Vite 构建成功
```

### 5.2 建议通过（视改动范围）

- 手动进关卡打一关（验证游戏可玩）
- 选关界面走一遍（验证 UI 不崩）
- 宗门/天命/天赋/皮肤/充值各点一遍（验证 meta 不崩）
- 关卡切换/档案切换（验证状态不泄漏）

---

## 6. 禁止事项（反模式）

| 禁止 | 原因 |
|---|---|
| 跳过 design review 直接写代码 | 架构债累积 |
| 实现后不更新文档 | 文档与代码脱节 |
| 新机制不写测试 | 纯函数不测 = 裸奔改数值 |
| 在引擎里硬编码新 behavior | 违反 OCP，应走策略注册表 |
| 直接改 Game.ts 超过 10 行而不抽模块 | 违反 SRP |
| 重复造"值对象"过度包装游戏数值 | YAGNI——`GridPos` 够用，别给每个 int 包一层 |

## 7. 实际案例（参考模式）

### 案例 A：加新敌人机制（减速/ice）

```
1. 设计：减速需要哪些数据字段？（TowerLevelConfig.slow / EnemyR.slowFactor）→ 写在内容设计中
2. Review：OCP 检查——现有 projectile 不受影响（slow 可选字段）✓；SRP——减速状态在 EnemyR，应用在 updateWave ✓
3. 文档：更新内容设计 §3.2 标注减速已实现
4. 实现：types.ts 加字段 → Game.ts updateWave/updateProjectiles 加逻辑 → Board.ts 加视觉
5. 验证：typecheck + 89 tests + build
```

### 案例 B：难度五星隔离

```
1. 设计：cleared key 改为 `${levelId}:${difficulty}`；isUnlocked 加 difficulty 参数 → 写在内容设计 §8.1
2. Review：SRP——isClearedOn/isUnlocked/recordResult 各司其职 ✓；OCP——加第 4 难度只需追加类型 ✓；迁移兼容旧存档 ✓
3. 文档：更新内容设计 §8.1 解锁规则描述
4. 实现：progress.ts 改签名→levelSelect.ts 改调用→withDefaults 迁移→测试更新
5. 验证：typecheck + 90 tests + build
```

## 8. 工作流检查表（每次开发前过一遍）

```
□ 1. 需求是否清晰？（要加什么、改什么）
□ 2. 是否需要新系统？（🟢配置 / 🟡引擎小改 / 🔴新子系统）
□ 3. 设计 Review 通过了吗？（五大可扩展性 check）
□ 4. ⚠️ **实现开始前：文档更新清单写了吗？**（§4.0——**写代码前先列**）
□ 5. SOLID 五条过完了吗？（SRP/OCP/LSP/ISP/DIP）
□ 6. 纯函数有测试吗？（新函数 + 受影响的老测试）
□ 7. **代码写完后：对照清单一笔一划划掉了吗？**（§4.0——**不允许跳过**）
□ 8. typecheck + test + build 全绿了吗？
□ 9. 旧文档引用删干净了吗？（重命名/删除文件后）
```
