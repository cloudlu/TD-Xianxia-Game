# 仙侠塔防 · 开发流程规范

> 定义本项目所有功能的开发流程。每步必须遵守，保证架构可扩展、代码可维护、文档与实现一致。

## 1. 总流程（六步法）

```
需求 → 设计 Review → 文档清单 → 实现 → 实现 Review → 合并前检查
  ↑                                                    │
  └────────────────────────────────────────────────────┘
```

**不可跳步**。设计未 review 不许写代码；实现 review 未通过不许合并。

**新增「实现 Review」和「合并前检查」两步**：历史教训——设计时想的 SOLID/测试/文档，实现后经常走样。必须有一个实现后的闭环校验，而非仅靠设计阶段的意图保证。

---

## 2. 设计阶段（按架构 skill）

### 2.1 触发条件
任何涉及以下之一的功能：
- 新系统（新机制/新敌人能力/新塔行为/新 UI 模块）
- 数据结构变更（Progression/LevelConfig/EnemyConfig/TowerConfig schema）
- 引擎改动（Game.ts / WaveManager / TowerOperations / TowerCombat）
- 平衡调整（塔数值/敌人数值/经济参数/章节缩放）

### 2.2 设计输出（写在文档里，不是口述）

| 内容 | 位置 |
|---|---|
| 系统架构描述 | 涉及哪些模块、新增哪些文件 |
| 配置 schema | 字段名/类型/默认值/示例 |
| 引擎改动点 | 具体到函数/方法/文件 |
| 数值公式 | 带演算验证 |
| 平衡论证 | 与现有系统的兼容性推演 |
| 实现复杂度 | 🟢配置/🟡引擎小改/🔴新子系统 |

### 2.3 Review 检查清单（五大可扩展性 + SOLID + 测试 + 文档）

```
[ ] 剧情可扩展：加章节是否只加数据？（不加代码）
[ ] 塔和敌人可扩展：数值类是否纯配置？新机制是否走 registry/strategy？
[ ] 平衡易调整：数值是否在配置文件中？是否提供了平衡旋钮？
[ ] 前后端可分离：数据访问是否走 Repository 接口？
[ ] 通用功能可扩展：加成是否走 Modifier 管线？
```

**新增——设计阶段必须同时评估以下三项**（以防设计 review 通过、实现后走样）：

```
[ ] SOLID 可行性：新增的类/模块 SRP 成立吗？OCP 成立吗？是否已有足够大的类需要先拆分？
[ ] 测试策略：新增模块有哪些测试？边界 case 是什么？(§3.3)
[ ] 受影响文档清单：实现后需要更新哪些文档？(§4.0)
```

### 2.4 发现的问题分级

- 🔴 **严重**：会返工/会崩平衡/承诺做不到。**不修不许进入实现**。
- 🟡 **重要**：影响完整性/体验。应修。
- 🟢 **次要**：建议。可后置。

## 3. 实现阶段（按 SOLID skill）

### 3.1 五条原则逐项过

| 原则 | 检查问题 |
|---|---|
| **S**RP 单一职责 | 这个类/模块只有一处变的理由吗？如果超过 300 行或有明显可分离的子职责 → 先拆再写。 |
| **O**CP 开闭原则 | 加新内容需要改已有代码吗？策略/注册表/配置就够了吗？ |
| **L**SP 里氏替换 | （本游戏以函数式为主，此条较少触发，但不制造隐藏契约） |
| **I**SP 接口隔离 | 接口是否只暴露调用方需要的方法？ |
| **D**IP 依赖倒置 | 引擎依赖抽象（ConfigLookup/CombatContext/SaveRepo）还是具体实现？ |

### 3.2 代码规范

```typescript
// ✅ 正确：配置驱动，纯函数
export function computeStars(lives: number, startLives: number): number { ... }

// ✅ 正确：策略注册表，不改 Game
class ProjectileStrategy implements AttackStrategy { ... }

// ✅ 正确：Repository 抽象
export interface SaveRepo { load(): Progression; save(p: Progression): void; }

// ❌ 错误：硬编码数值
if (towerId === 'flying_sword') { dmg = 20; }

// ❌ 错误：向引擎散落新逻辑
// Game.fire() 里加 if (behavior === 'new_type') { ... }
```

### 3.3 测试要求

- **所有新增模块/类必须有独立测试文件**（纯函数、状态类、注册表类、UI 组件，一视同仁）。
  - 纯函数：测输入输出边界（正常/异常/空/极值）。
  - 状态类（如 WaveManager/TowerOperations/TowerCombat）：测构造、主入口方法、边界条件、错误路径、状态收敛。
  - 注册表类：测注册/查询/缺省/重复。
  - UI 组件：测渲染结果、交互响应。
- 配置变动必须同步更新测试预期值。
- 运行 `npm test` 全绿后方可进入验证阶段。
- ❌ 禁止以"这个类太简单不需要测"为由跳过——简单类测的就是"没写错"。

### 3.4 实现中自检（新增）

实现过程中每次 commit 前问自己：

```
□ 我新增的类是否超过 250 行？是否需要进一步拆分？
□ 新类/模块有对应的 .test.ts 吗？覆盖了正常+边界+错误路径吗？
□ 我改动的旧模块，其现有测试还能通过吗？
□ 设计阶段写的「受影响文档清单」开始处理了吗？（至少每实现一个模块标记一个已同步）
```

## 4. 文档更新（不可跳步）

### 4.0 ⚠️ 必须——实现开始前立即写下需更新的文档清单

**在写第一行代码之前**，根据改动范围列出"需要更新哪些文档"并记在对话中。**同时预估每项更新所需的时间**（如 +5min/项），确保文档更新不被安排在"最后 5 分钟"。实现完成后对照清单逐条划掉。不允许 typecheck 通过就直接认为完成。

示例：
```
改动：talents.ts 新增 fire_mastery + thunder_mastery；equipment.ts spirit_flame 改 stat
→ 需更新文档：[ ] 装备与皮肤设计.md（装备表/stat 名变更，+5min）
               [ ] 设计文档.md（如有提到天赋数量，+2min）
```

### 4.1 更新规则

| 文档 | 什么情况下必须更新 |
|---|---|
| **设计文档.md** | 新增/变更系统架构、机制、平衡公式；实现现状小节 |
| **内容设计.md** | 新增/变更章节、关卡、敌人、塔、叙事 |
| **装备与皮肤设计.md** | 新增/变更装备、皮肤词条 |
| **README.md** | 影响运行方式、部署方式、核心玩法 |
| **TODO.md** | 完成项从待办移至"已完成" |

### 4.2 不得出现的情况

- ❌ 代码实现了但文档还写"未实现"
- ❌ 数值改了但文档公式是旧值
- ❌ 删了敌人/塔但文档表格还列着
- ❌ 文档里引用了已删除的文件名

### 4.3 文档同步验收（新增）

**实现完成后**，由 review 者（或自己一天后回看）逐条核对 4.0 的文档清单：

```
[ ] 清单上的每项文档都已更新？
[ ] 新模块的路径/文件名在文档中体现？
[ ] 旧模块的代码引用（文件路径/类名）已更新？
[ ] 设计文档的「实现现状」小节已追加当前变更说明？
```

## 5. 实现 Review 阶段（新增——三步必过）

设计 review 通过只代表"方案可行"——实现走样是常见问题。在合并前增加一个轻量实现 review。

### 5.1 步骤一：架构自检

对照 §3.1 SOLID 逐条过**实际的代码**（而非设计阶段的意图）：

```
□ SRP：每个类/模块真的只有一处变的理由吗？有没有悄悄塞进第二个职责？
□ OCP：新增行为是否需要改已有代码？还是只加文件/注册就行？
□ DIP：依赖是接口还是具体类？引擎是否依赖了 UI 层？
□ 如果有提取新类的迹象（超过 200 行的私有方法、多个可分离的职责集合），是否真的拆了？
```

### 5.2 步骤二：测试自检

```
□ 新模块的 test 文件覆盖率 = 构造 + 正常路径 + 边界条件 + 错误路径
□ 旧模块的现有测试是否全部通过？（运行 npm test）
□ 如果改动导致旧测试预期值变化——是真的优化了行为，还是不小心改了？（确认后更新预期值）
```

### 5.3 步骤三：文档自检

```
□ 逐条核对 §4.0 的文档更新清单——全部划掉了吗？
□ 设计文档「实现现状」或修订记录更新了吗？
□ 删了/重命名了文件，其他文件对旧路径的 import 更新了吗？
```

### 5.4 审查者责任

实现 review 可以由原作者自检，但**含有以下任何一条时，必须由第二人审查**：

- 涉及存档 schema 变更
- 涉及经济/平衡核心公式
- 新增超过 300 行的新文件
- 重构涉及 5 个以上文件

## 6. 验证阶段

### 6.1 必须通过

```
npm run typecheck    # TypeScript 零错误
npm test             # 全部单元测试通过
npm run build        # Vite 构建成功
```

### 6.2 建议通过（视改动范围）

- 手动进关卡打一关（验证游戏可玩）
- 选关界面走一遍（验证 UI 不崩）
- 宗门/天命/天赋/皮肤/充值各点一遍（验证 meta 不崩）
- 关卡切换/档案切换（验证状态不泄漏）

---

## 7. 禁止事项（反模式）

| 禁止 | 原因 |
|---|---|
| 跳过 design review 直接写代码 | 架构债累积 |
| 实现后不更新文档 | 文档与代码脱节 |
| 新机制/新模块不写测试 | 代码不可验证 = 裸奔 |
| 在引擎里硬编码新 behavior | 违反 OCP，应走策略注册表 |
| 直接改 Game.ts / 大型文件超过 10 行而不抽模块 | 违反 SRP |
| 重复造"值对象"过度包装游戏数值 | YAGNI——`GridPos` 够用，别给每个 int 包一层 |
| **设计 review 考虑了 SOLID/测试/文档，实现后不核对直接合并** | 方案与结果脱节——本流程新增 §5 就是为了防止这个 |

## 8. 实际案例（参考模式）

### 案例 A：加新敌人机制（减速/ice）

```
1. 设计：减速需要哪些数据字段？（TowerLevelConfig.slow / EnemyR.slowFactor）→ 写在内容设计中
2. Review：OCP 检查——现有 projectile 不受影响（slow 可选字段）✓；SRP——减速状态在 EnemyR，应用在 updateWave ✓；测试：测 slow 字段读写 + 减速实际生效；文档：更新内容设计 §3.2 标注减速已实现
3. 文档清单：内容设计.md（标注实现）+ 设计文档.md（如有提到减速机制）
4. 实现：types.ts 加字段 → Game.ts updateWave/updateProjectiles 加逻辑 → Board.ts 加视觉
5. 实现 Review：SRP——新逻辑是否塞进了已有大函数？需不需要提前抽模块？测试——加了减速的 projectile 测试和减速结算测试？文档——内容设计 §3.2 更新了？
6. 验证：typecheck + all tests + build
```

### 案例 B：难度五星隔离

```
1. 设计：cleared key 改为 `${levelId}:${difficulty}`；isUnlocked 加 difficulty 参数 → 写在内容设计 §8.1
2. Review：SRP——isClearedOn/isUnlocked/recordResult 各司其职 ✓；OCP——加第 4 难度只需追加类型 ✓；迁移兼容旧存档 ✓
3. 文档清单：内容设计.md（解锁规则）+ 设计文档.md（如有难度相关描述）
4. 实现：progress.ts 改签名→levelSelect.ts 改调用→withDefaults 迁移→测试更新
5. 实现 Review：存档 schema 变更→需第二人审查（§5.4）。测试——旧存档迁移测试加了吗？文档——解锁规则描述更新了？
6. 验证：typecheck + 90 tests + build
```

### 案例 C：提取引擎模块（SRP 拆分，本轮教训总结）

```
1. 设计：Game.ts 562 行负责 6+ 职责 → 拆出 WaveManager（波次）/ TowerOperations（塔操作）/ TowerCombat（战斗）
2. Review：SRP——每新类一个职责 ✓；可扩展性——达标 ✓；测试——每个新模块需独立测试；文档——§1.3 目录结构需更新 + 实现现状加 v0.73 说明
3. 文档清单：设计文档.md（§1.3 目录结构 + 实现现状 v0.73 追加）
4. 实现：逐模块提取，保持 Game.ts 对外接口不变
5. **实现 Review**（本流程新增后必须执行）：
   - SOLID：提取后的类 SRP 成立？Game.ts 是否真的瘦了？→ ✅
   - 测试：WaveManager.test.ts / TowerOperations.test.ts / TowerCombat.test.ts 写了？覆盖边界？→ ✅
   - 文档：设计文档 §1.3 的旧文件名（EntityManager/CombatSystem/Targeting）更新为实际文件名？→ ✅
6. 验证：typecheck + 191 tests + build → ✅
```

## 9. 工作流检查表（每次开发前过一遍）

```
□ 1. 需求是否清晰？（要加什么、改什么）
□ 2. 是否需要新系统？（🟢配置 / 🟡引擎小改 / 🔴新子系统）
□ 3. 设计 Review 通过了吗？（五大可扩展性 + SOLID 可行性 + 测试策略 + 文档清单）
□ 4. ⚠️ **实现开始前：文档更新清单写了吗？**（§4.0——**写代码前先列**）
□ 5. 实现中自检（§3.4）：每 commit 前过了吗？
□ 6. **实现 Review**（§5）：架构自检 + 测试自检 + 文档自检 + 需要第二人审查吗？
□ 7. typecheck + test + build 全绿了吗？
□ 8. 旧文档引用删干净了吗？（重命名/删除文件后）
□ 9. §4.0 的文档清单完全划掉了吗？（最终确认）
```

**对比旧版的变更**（v0.1 → v0.2）：
- 新增「实现 Review」阶段（§5），防设计/实现脱节
- 新增 §3.4 实现中自检清单，防积累到最后一团乱
- 新增 §4.3 文档同步验收，防文档清单列了不执行
- 新增 §5.4 第二人审查条件
- 设计 review 清单新增 SOLID/测试/文档三项（§2.3）
- 测试要求扩展到所有模块（不再仅限于纯函数，§3.3）
- 案例 C 新增 SRP 拆分参考模式（§8）
- 工作流检查表新增 §4/§5 相关项（§9）
- 总流程 5 步→6 步
